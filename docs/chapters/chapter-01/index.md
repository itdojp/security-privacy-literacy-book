# 第1章：セキュリティ事故の典型と影響（なぜ守るのか）

## この章で学ぶこと

- 典型事故（誤公開/漏えい/権限事故）のパターンを把握する
- 影響（顧客/法令/信用/復旧コスト）を説明できる
- 予防のためにチェックリスト運用が必要な理由を理解する

## 成果物（または判断基準）

- 事故パターンと予防策の対応表
- 付録チェックリストへの導線（[チェックリスト集](../../appendices/checklists/)）

## 本文

> 注意: 本書は一般的な実務ガイドであり、法務助言ではありません。組織のポリシー/法令/契約/顧客要件が優先されます。相違がある場合は、情報セキュリティ/法務へ確認してください。

事故は“技術的ミス”だけでなく、“確認の抜け”から発生する。新人の段階では、個別最適の知識よりも、典型パターンと確認の型を先に持つ。

### 典型パターン（例）

- 誤公開: ストレージ/リポジトリ/設定の公開範囲ミス
- 漏えい: ログ/Issue/CI 出力への秘密情報混入
- 権限事故: 過剰権限、退職/異動後の残権限

まずは“どの事故を防ぐか”を言語化し、チェック項目に落とす。

## 具体例（悪い例→良い例）

### 悪い例

```md
大丈夫だと思います。
公開設定の確認はしていません。
```

### 良い例

```md
リリース前チェックを実施（付録のチェックリスト）
公開範囲/IAM/ログの項目を確認し、結果を記録
```

## チェックリスト

- [ ] 典型事故パターンを説明できる
- [ ] 影響（誰に/何が/どれくらい）を説明できる
- [ ] チェックリストに落とせる粒度で予防策がある

## まとめ

- 事故は“知っていた”では防げないため、チェックリストで運用に落とす
- 秘密情報/権限/公開設定は、レビューと棚卸しで継続的に守る
